## Мета роботи  
Дослідити модель паралельного програмування з розподіленою пам’яттю, принципи обміну повідомленнями між процесами (Message Passing), а також порівняти її зі спільною пам’яттю на прикладі проблем race condition та deadlock.

---

## Теоретичні відомості  

У паралельних системах виникають специфічні проблеми синхронізації:

Race Condition — ситуація, коли результат виконання залежить від порядку доступу потоків до спільних даних.

Deadlock — взаємне блокування потоків, коли кожен очікує ресурс, зайнятий іншим.

Starvation — потік не отримує доступ до ресурсів через постійне блокування іншими потоками.

Livelock — потоки активно змінюють стан, але не виконують корисної роботи.

---

## Моделі пам’яті

### Shared Memory (спільна пам’ять)
Усі потоки працюють із загальними даними. Для синхронізації використовується lock (mutex), що гарантує атомарність критичних секцій.

### Message Passing (розподілена пам’ять)
Процеси не мають спільної пам’яті та взаємодіють через передачу повідомлень. У роботі використано:
- Queue (черга повідомлень)
- Pipe (двонаправлений канал)
- Shared Value з Lock (гібридний механізм)

Message Passing реалізовано через multiprocessing Python, що імітує базові принципи обміну повідомленнями у розподілених системах (send/receive).

## Постановка задач

### Задача 1: Race Condition та Deadlock

Модель банківської системи:

- понад 100 рахунків із початковим балансом  
- понад 1000 паралельних транзакцій  
- випадкові перекази між рахунками  

Реалізовано три версії:

Sequential — послідовне виконання без паралельності  
Race Condition — без синхронізації (демонстрація помилки)  
Safe Version — використання lock для усунення race condition  

Deadlock змодельовано через конкуренцію за ресурси та уникнуто за допомогою timeout-підходу.

---

### Задача 2: Message Passing

Реалізовано міжпроцесну взаємодію:

- Queue — обмін через чергу повідомлень  
- Pipe — двонаправлений канал передачі даних  
- Shared Memory — спільне значення з синхронізацією  

Основний процес генерує дані, передає їх у worker-процес та отримує результат назад.

---

### Додатково: Cross-language IPC

Реалізовано взаємодію Python ↔ C++ через subprocess:

- Python передає дані через stdin  
- C++ обробляє та повертає результат через stdout

## Аналіз результатів

- Race condition дає некоректні результати через відсутність синхронізації  
- Lock повністю усуває race condition, але додає overhead  
- Deadlock уникнуто через timeout та контроль ресурсів  
- Message Passing повільніший, але безпечніший  
- Shared memory найшвидший, але найризикованіший
